## Key Interactions to Verify

### Worker → Filho transitions
- Spawn → `agent_state: spawning` no JSON local
- Ack → `agent_state: working`
- Heartbeat periódico → atomic write tempfile+rename
- Done → bd status `awaiting_integration` (D2=A: bd primeiro) + JSON `agent_state: done`
- Watchdog timeout → `agent_state: stuck` → `IssueStatus: blocked` se cap atingido

### MergeManager → Épico transitions
- Detect "todos filhos awaiting_integration" via `bd list --parent=... --status=awaiting_integration` (single query, D14=A)
- Single-flight lock (D11=A) — apenas 1 goroutine dispara merger por épico
- Spawn merger agent → take loop reusado
- Parse `merge_outcome:` da description após agent done (D6=A enum exaustivo)
- Roteamento: `success` → épico `awaiting_pr_review`; `merge_conflict`/`push_failed`/`pr_create_failed` → `blocked`; `pr_already_exists` → adopta PR e vai pra `awaiting_pr_review`

### Sweep → fechamento de épico
- Cache MERGED hit no 2º tick (D5=C) — não re-consulta `gh pr view`
- Circuit breaker abre após 3 falhas consecutivas; backoff 5/15/60min
- PR stale WARN >N dias (TT2=B) sem ação automática
- `bd close` filhos + épico idempotente

### `kernl epic merge <id>` (D4=A)
- Validação: épico em `blocked` com `merge_conflict_at` setado
- Validação: filhos em `awaiting_integration` ou `closed`
- Re-dispatch idempotente — re-rodar em estado errado retorna erro claro

## Edge Cases

### `workflow/agent_state_store.go`
- Arquivo ausente → defaults + WARN (D9=A)
- JSON corrompido → defaults + WARN
- Disk full em write → erro propaga
- Concurrent writes no mesmo bead (mesmo processo) → mutex serializa
- Permissions errado em `~/.kernl/state/` → erro descritivo
- Versão futura do schema (sem `schema_version` no MVP) → tolerância via best-effort parse + WARN

### `workflow/ensure_custom.go` (D7=B)
- bd fresh sem customs → registra ambos (`awaiting_integration`, `awaiting_pr_review`)
- Customs já set → no-op (cache hit)
- Customs estrangeiros presentes (ex: outro consumer registrou `awaiting_qa`) → preserva, faz merge
- Customs do kernl foram removidos por terceiro → re-adiciona
- `bd config set` falha (binary missing, permissions) → fail-loud no startup
- Sentinel staleness (mtime do `.beads/.bd-config` mudou) → re-validate

### `workflow/description.go`
- Linhas com `:` no valor (ex: `pr_url: https://github.com/...`) — preservar valor inteiro
- BOM no início do texto → strip
- Empty description → addMetadataField cria estrutura mínima
- Linha duplicada do mesmo key → addMetadataField faz update da primeira, remove demais
- Multilinha (valor com `\n`) → não suportado no MVP (assert single-line)

### `merge/manager.go` trigger
- 0 filhos no épico → trigger nunca dispara (no-op); spec edge case
- 1 filho em `blocked` → halt épico, NÃO disparar merger
- N filhos awaiting_integration + 1 still in_progress → wait, não disparar
- Race: último filho transitiona durante scan → single-flight lock cobre (D11=A)
- Re-trigger após restart do kernl serve → bd é SoT (D2=A), trigger re-detecta corretamente

### `merge/manager.go` outcome parse
- `merge_outcome: success` → épico `awaiting_pr_review`
- `merge_outcome: merge_conflict` → blocked + `merge_conflict_at:` na description
- `merge_outcome: push_failed` → blocked
- `merge_outcome: pr_create_failed` → blocked
- `merge_outcome: pr_already_exists` → busca pr_url existente, adopta, vai pra `awaiting_pr_review`
- Outcome ausente (agent não escreveu) → blocked com diag "merger agent did not report outcome"
- Outcome inválido (string desconhecida) → blocked com diag "unknown merge_outcome: X"

### `sweep/sweep.go`
- 0 épicos em `awaiting_pr_review` → tick silent (D13=A: skip empty logs)
- `gh pr view` retorna 401 (token expirado) → breaker.fail + WARN
- `gh` binary missing → fail-loud no startup (não silent)
- PR state == OPEN → no-op, mas PR stale check (TT2=B)
- PR state == CLOSED (não MERGED) → diag WARN; idealmente revisar a decisão de marcar épico, mas no MVP só WARN
- PR state == MERGED → bd close + cache; idempotente em re-tick
- Cache de PR MERGED sobrevive a restart? NÃO (memória) — re-popula no 1º tick pós-restart

### `cmd/kernl/sweep.go`
- `--dry-run`: zero writes ao bd; output mostra "would close N child beads + epic"
- Sem flag: efetua bd close; output mostra "closed N child beads + epic"

## Critical Paths (E2E)

### Path 1 — Happy: épico de 3 filhos paralelos, sem conflito
```
GIVEN
  bd inicializado, epic kernl-abc criado com 3 filhos paralelos (sem deps entre si)
  workers configurados, gh autenticado, repo limpo

WHEN
  kernl epic run kernl-abc

THEN
  1. Epic branch feat/kernl-abc criada a partir de master
  2. 3 worktrees criadas, 3 workers spawned em paralelo
  3. Filhos transitam open → in_progress → awaiting_integration
  4. EpicExecutor detecta "todos awaiting_integration" e dispara merger
  5. Merger agent: cd epic worktree, git merge --no-ff dos 3 filhos em ordem topológica
  6. git push origin feat/kernl-abc
  7. gh pr create → recebe URL
  8. agent escreve merge_outcome: success + pr_url: ... na description do épico
  9. MergeManager: filhos viram closed, épico vira awaiting_pr_review
  10. (humano aprova/mergeia PR no GitHub)
  11. kernl sweep detecta MERGED → bd close épico + filhos (idempotente)
```

### Path 2 — Conflict + manual resolve + re-dispatch
```
GIVEN
  Mesmo setup, mas filhos 1 e 2 editam o mesmo arquivo

WHEN
  kernl epic run kernl-abc

THEN
  1-4. Como Path 1
  5. Merger tenta git merge filho-2 → conflito
  6. Agent escreve merge_outcome: merge_conflict + merge_conflict_at: feat/kernl-def
  7. MergeManager: épico vira blocked
  8. (humano: cd ~/.kernl/worktrees/kernl-abc/, resolve conflito, commit)
  9. (humano: kernl epic merge kernl-abc)
  10. cmd valida épico em blocked com merge_conflict_at OK
  11. cmd valida filhos awaiting_integration OK
  12. cmd marca épico in_progress, re-dispatch merger
  13. Merger continua de onde parou (próximo filho ou push)
  14. Path normal: success → awaiting_pr_review
```

### Path 3 — Push fail → blocked → recover
```
GIVEN
  Setup com gh autenticado mas branch protection regra rejeitando push direto pra feat/

WHEN
  kernl epic run kernl-abc

THEN
  1-5. Como Path 1
  6. git push falha (rejected by branch protection)
  7. Agent retry 2x com backoff → ainda falha
  8. Agent escreve merge_outcome: push_failed
  9. MergeManager: épico blocked
  10. (humano: ajusta branch protection ou pusha manualmente)
  11. (humano: kernl epic merge kernl-abc)
  12. Re-dispatch: merger detecta merges já feitos, tenta push de novo
  13. Sucesso → gh pr create → awaiting_pr_review
```

## Regression Tests (mandatórios — IRON RULE)

### R1 — Passo A integration test verde contra bd 1.0.4 real
- Setup: `bd init --from-jsonl` no harness (já consertado em `fix/integration-test-setup`)
- Cria 1 bead, kernl claim, kernl spawna worker, worker reporta done
- Asserts: `bd show <id>` retorna status `awaiting_integration` SEM `validation failed`
- **Critério #1 da spec.** Sem isso, MVP não está validado.

### R2 — 962 unit tests passam pós-migração
- Cada `*_test.go` com string literal legacy refatorado pra constante de `workflow/`
- Suite verde como gate de pre-merge

### R3 — Fixtures migrated cleanly
- `testdata/beads-*/.beads/issues.jsonl` aplicado o `sed` map per spec §8.4:
  - `ready_for_implementation` → `open`
  - `implementation` → `in_progress` (com `agent_state: working` no JSON local se teste depender)
  - `implementation_review` → `awaiting_integration`
  - `ready_for_shipment` → `awaiting_integration`
  - `shipment` / `shipment_review` / `shipped` → `closed`
  - `deferred` → `closed` com `reason: "deferred"` (preserva semantic legacy)
  - `abandoned` → `closed` com `reason: "abandoned"`
- Tests que asserts especificamente sobre distinção foolery entre `deferred` e `closed` → deletados (conceito sai do modelo)

## Notas operacionais

- Test env requer **bd CLI ≥ 1.0.4** (D10=A) — pinado no harness com fail-fast.
- Test env requer **gh CLI** disponível mas pode usar GH_TOKEN fake; testes E2E que precisam de PR real usam repo de fixture.
- `~/.kernl/state/` em test env aponta pra tmpdir per-test pra evitar cross-test contamination.
- Watchdog tolerância deve ser configurável por test (default produção curto pode flake em CI lento) — sugerir env var `KERNL_WATCHDOG_DISABLE=1` pra unit tests que não estão testando watchdog.
